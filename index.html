7 habits of good cooks

Writing code you can trust

tip 1

Run code in chunks

Break code into small pieces that are easy to run and read. #%% makes a runnable cell inside a normal .py file (needs the Jupyter extension).

interactive mode

Run cells right inside a .py file

A #%% comment turns the next chunk into a runnable cell. Click Run Cell and results (and plots!) appear in an interactive window beside your script — no file saved, running in your current conda env.

▷ Run Cell
#%% load the data
import numpy as np
data = np.loadtxt("emg.csv")
 
▷ Run Cell
#%% inspect
print("mean:", data.mean())

tip 2

Write pseudocode first

Design in plain English before you code: inputs, outputs, what's computed, which variables, any loops or logic.

# INPUT: list of signal values
# OUTPUT: indices where signal > threshold
# for each value:
#     if value > threshold:
#         remember its index
# return the remembered indices
Pseudocode is the recipe outline. Turning it into Python is just translation.
tip 3

Comment your code (for future-you)

❌ Not helpful:

x = x + 1  # add one to x

✅ Helpful:

peaks += 1  # count another heartbeat
▶ live tip 4

Don't reinvent the wheel

Common libraries already solved the hard parts. Use them.

▶ live tip 5

Don't repeat yourself — write functions

If a chunk of code is "a step in a process," make it a function and reuse it.

▶ live tip 6

Read your error messages

Run this — the traceback tells you where and why. Then fix line 3 (compare numbers, not a string).

▶ live tip 7

Test your code — check the extremes

GitHub

Sharing recipes with the other restaurants.

How do you share a recipe and its history?

  • Tell it face to face? — lost the moment you walk away.
  • Hand over a photo? — hard to update or send back.
  • Keep the recipe online, versioned, with notes on every change. ✅

GitHub hosts your code; git is the version-control system that tracks changes.

  3. Partner reviews & comments; once approved, merge into main.
Only merge when code is "done." And don't delete your personal branch.

the workflow

How we'll use GitHub in this class

📁One repo per project, shared with your partner
🌿Each partner works on their own branch
🐍Code in your own .py files, pushed often
📓Final notebook imports functions from those .py files
Regular pull requests merge your branch into main — small and often beats one giant merge.

The protocol for every coding session

1️⃣pull from remote
2️⃣open folder & code + save
3️⃣commit + push to remote
4️⃣when happy, open a pull request

🗣️ …and talk to your partner when you're both coding.

watch out

When GitHub gets messy

ProblemAdvice
Same file, same time, two peopleDon't. Split files, or push/pull back and forth. Talk first.
Two files with the same namePut your name in your filenames.
Files you don't want trackedAdd them to .gitignore.
File over 100 MB.gitignore it; share big data via Box.
Broke working code, can't get backTest in a fresh file / branch; worst case git reset (ask for help).

A notebook (.ipynb) mixes markdown, code, and output — so a reader can step through your analysis like a report.

📓a .ipynb file — code + prose + output in one
🧩needs the Jupyter extension (you installed it)
📦and the ipykernel package in your env
